# Document Summarizer Skill - 使い方ガイド

長文ドキュメントを効率的に要約し、重要なポイントを抽出するスキルです。

## クイックスタート

### 1. 基本的な使い方

Claude Codeで以下のようにリクエストします：

```
以下のドキュメントを要約してください：

[ドキュメントの内容を貼り付け]

要約レベル: 標準
```

### 2. ファイルから要約

```
docs/architecture.md を要約してください。
要点のみお願いします。
```

### 3. 高度な要約

```
README.md を詳細に要約:
- 出力形式: エグゼクティブサマリー + 箇条書き
- 焦点: インストール手順、主要機能
- 対象読者: 初めて使う開発者
```

## 要約レベル

### レベル1: 要点のみ (Brief)

**特徴**: 最も簡潔、2-5個の箇条書き

**使用例**:
```
この技術文書の要点のみ教えてください。
読了時間: 30秒以内
```

**出力例**:
```
• Docker は軽量なコンテナ仮想化技術
• アプリケーションと依存関係をパッケージ化
• 環境の一貫性を保証し、デプロイを簡素化
```

### レベル2: 標準 (Standard)

**特徴**: バランスの取れた要約、1-3段落または5-10箇条書き

**使用例**:
```
このドキュメントを標準レベルで要約してください。
```

**出力例**:
```
## 概要
Docker は、アプリケーションをコンテナという軽量な仮想環境で
実行する技術です。従来の仮想マシンと比べて起動が速く、
リソース効率が高いのが特徴です。

## 主要な機能
• コンテナイメージ: アプリケーションと依存関係を含むパッケージ
• Docker Hub: イメージの共有リポジトリ
• Docker Compose: 複数コンテナの管理
• オーケストレーション: Kubernetes との統合

## 利点
開発、テスト、本番環境で同じコンテナを使用することで、
「私の環境では動く」問題を解決し、デプロイを大幅に簡素化します。
```

### レベル3: 詳細 (Detailed)

**特徴**: 包括的な要約、複数セクション

**使用例**:
```
詳細な要約をお願いします。
すべての重要なセクションを含めてください。
```

**出力例**:
```
[複数セクションにわたる詳細な要約]
- エグゼクティブサマリー
- 背景と目的
- 技術詳細
- 使用方法
- ベストプラクティス
- トラブルシューティング
- まとめと推奨事項
```

## 出力形式

### 1. 箇条書き

**リクエスト**:
```
箇条書き形式で要約してください。
```

**出力**:
```
## 主要なポイント

• **セキュリティ**
  - HTTPS必須
  - 認証トークンの有効期限: 1時間
  - レート制限: 1000リクエスト/時

• **パフォーマンス**
  - 平均レスポンス: 50ms以下
  - 最大同時接続: 10,000
  - キャッシュ戦略: Redis使用
```

### 2. 段落形式

**リクエスト**:
```
段落形式でレポートスタイルの要約をお願いします。
```

**出力**:
```
本システムのセキュリティ対策として、すべての通信でHTTPSを必須としています。
認証トークンの有効期限は1時間に設定され、レート制限により
1時間あたり1000リクエストに制限されています。

パフォーマンス面では、平均レスポンスタイムを50ms以下に保ち、
最大10,000の同時接続に対応しています。Redisを使用したキャッシュ戦略により、
高速なデータアクセスを実現しています。
```

## 高度な使い方

### 焦点の指定

特定のトピックに集中した要約:

```
セキュリティとパフォーマンスに焦点を当てて要約してください。
その他の詳細は省略してOKです。
```

### 対象読者の指定

読者に合わせた要約:

```
非技術者の経営陣向けに要約:
- 専門用語は避けるか説明を追加
- ビジネスインパクトを強調
- 技術詳細は最小限に
```

### 長さの制限

```
500文字以内で要約してください。
```

または

```
5つの箇条書きにまとめてください。
```

### 複数ドキュメントの統合要約

```
以下の3つの関連文書を統合して要約:
1. 技術仕様書
2. ユーザーマニュアル
3. リリースノート

出力:
- 全体概要（1段落）
- ドキュメント別の要点（各3-5ポイント）
- 統合的な推奨事項
```

### セクション別要約

長文を分割して要約:

```
この200ページのドキュメントを章ごとに要約:
- 各章: 3-5個の箇条書き
- 最後に全体サマリー
```

### コード要約

```
以下のPythonコードを要約:
[コードを貼り付け]

要約内容:
• 目的と機能
• 主要なクラス/関数
• 使用しているライブラリ
• 設計パターン
• 改善提案があれば
```

## HTMLレポート生成

美しくフォーマットされたHTML要約レポートを生成:

```
HTMLレポート形式で要約を生成してください:
タイトル: "技術仕様書要約レポート"
テーマ: プロフェッショナル（青系）
含める:
- 目次
- セクション別要約
- ハイライトボックス
- 印刷最適化
```

生成されたHTMLレポートの特徴:
- プロフェッショナルなデザイン
- 印刷に最適化
- セクションの色分け
- 重要度のビジュアル表示
- PDFエクスポート可能

**シナリオ**: 300ページのAPI仕様書を素早く理解したい

**リクエスト**:
```
この API仕様書を要約:
要約レベル: 標準
焦点:

- 認証方式
- レート制限
- エラーハンドリング

出力形式: 箇条書き

```

### 例2: 会議前の資料予習

**シナリオ**: 明日の会議資料（3つのPDF）を短時間で把握

**リクエスト**:
```
以下の3つの会議資料を各々要点のみで要約:
1. 売上レポート.pdf
2. 市場分析.pdf
3. 提案書.pdf

各資料: 3つの箇条書きまで
全体: A4用紙1枚に収まる程度
```

### 例3: 論文レビュー

**シナリオ**: 同じトピックの論文5本を比較レビュー

**リクエスト**:
```
機械学習に関する以下の5論文を要約・比較:
[論文リスト]

各論文: 標準レベルで要約
最後に: 比較表を作成（手法、結果、貢献）
```

### 例4: コードベースの理解

**シナリオ**: 初めて見るGitHubリポジトリを理解

**リクエスト**:
```
このリポジトリのREADMEとコードを要約:
焦点:
- プロジェクトの目的
- アーキテクチャ概要
- セットアップ手順（簡略版）
- 主要なコンポーネント
- 開発に必要な前提知識

```

### 例5: 契約書の確認

**シナリオ**: SaaS契約書の主要条項を確認

**リクエスト**:
```
この利用規約を要約:
焦点:
- 主要な権利と義務
- 料金体系
- データ取り扱い
- 解約条件
- 重要な制限事項

注意: 法的判断は専門家に確認してください
```

## ベストプラクティス

### 1. 明確な目的を設定

```
❌ 悪い例:
「このドキュメントを要約して」

✅ 良い例:
「この技術文書を要約してください。
対象: 新しいチームメンバー
目的: プロジェクトの全体像把握
焦点: アーキテクチャ、主要技術、セットアップ手順
読了目標: 10分以内」
```

### 2. 適切な要約レベルを選択

- **緊急度高** → 要点のみ
- **初見のドキュメント** → 標準
- **深い理解が必要** → 詳細

### 3. 出力形式を使い分ける

- **メール・チャット共有** → 箇条書き
- **レポート作成** → 段落形式
- **経営層への報告** → エグゼクティブサマリー
- **FAQ作成** → Q&A形式

### 4. コンテキストを提供

要約の精度を上げるために以下を指定:
- ドキュメントの種類
- 対象読者
- 使用目的
- 重要な焦点

### 5. 要約後の確認

生成された要約を確認し、重要な情報が漏れていないかチェック。
特に数値、日付、固有名詞は注意。

## トラブルシューティング

### 要約が長すぎる

**解決策**:
```
もっと簡潔にお願いします。
最大5つの箇条書きに収めてください。
```

### 重要な情報が抜けている

**解決策**:
```
以下のポイントを含めて再度要約してください:
- [抜けているポイント1]
- [抜けているポイント2]
```

### 専門用語が多すぎる

**解決策**:
```
非技術者向けに要約し直してください。
専門用語は避けるか、簡単な説明を追加してください。
```

### 焦点がずれている

**解決策**:
```
セキュリティとパフォーマンスに焦点を絞って要約してください。
その他の詳細は省略してOKです。
```

## 参考: 要約品質チェックリスト

生成された要約をチェック:

- [ ] 主要なポイントがすべて含まれているか
- [ ] 対象読者に適した表現か
- [ ] 事実関係に誤りはないか
- [ ] 数値、日付、固有名詞は正確か
- [ ] 適切な長さか（長すぎ/短すぎないか）
- [ ] 構造が論理的か
- [ ] アクションアイテムが明確か（必要な場合）
